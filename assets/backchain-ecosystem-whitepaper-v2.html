Section 1

Overview

Backchain is an unstoppable DeFi ecosystem built on Arbitrum (Ethereum L2). The protocol consists of 16 interconnected Solidity smart contracts compiled with Solidity v0.8.28, organized in a Hub & Spoke architecture designed for infinite extensibility.

  • Immutable and permissionless — no admin keys, no pause functions, no blacklists. Code is law.
  • Hub & Spoke design — a central orchestrator connects all modules; new spokes can be added without modifying existing contracts.
  • Activity-based mining — using the platform is mining. Every interaction generates fees that convert to BKC rewards.
  • Operator system — permissionless developer commissions. Any builder can earn by passing their wallet address as the operator parameter.
  • Vanilla JS frontend — no framework, no React, no Vue. Pure ES modules with Hardhat for contracts and Vercel for hosting.
Philosophy: Backchain is designed as unstoppable, permissionless DeFi infrastructure. All contracts support the Operator System where third-party builders earn commissions simply by passing their wallet address into every transaction.
Section 2

Hub & Spoke Architecture

2.1 — The Hub: BackchainEcosystem

Contract: BackchainEcosystem.sol (formerly EcosystemManager)

The Hub is the central orchestrator for the entire ecosystem. Every spoke contract sends its fees to the Hub, which handles distribution according to per-module configurations. This design allows new modules to be added without modifying any existing contract.

  • Manages per-module fee configurations — operatorBps, treasuryBps, buybackBps, customBps
  • Collects all ETH fees via collectFee(moduleId, operator)
  • Automatic fee distribution — splits to operator, tutor, buyback, treasury, and custom destinations
  • Tutor referral system — stores tutor-student relationships on-chain
  • Operator system — permissionless developer commissions
  • pendingEth[address] tracks withdrawable operator earnings

2.2 — The Spokes: 15 Module Contracts

Every spoke generates fees that flow back to the Hub. The spoke contracts are fully independent — they only need a reference to the Hub address. New spokes can be deployed and registered at any time without modifying existing contracts.

[ Any Spoke Contract ]
↓ calls collectFee(moduleId, operator)
[ BackchainEcosystem (Hub) ]
↓ ↓ ↓ ↓ ↓
Tutor  |  Operator  |  BuybackMiner  |  Treasury  |  Custom

Section 4

Fee System Architecture

Every protocol action calls BackchainEcosystem.collectFee(moduleId, operator). Fees are split per module configuration with a two-stage waterfall:

Stage 1 — Tutor Cut

10% off the top goes to the user's Tutor (referrer). If the user has no tutor, this 10% is burned instead. This creates a strong incentive for community growth.

Stage 2 — Remaining 90% Split

The remaining 90% is distributed according to each module's immutable configuration:

  • operatorBps → Operator wallet (or burned if no operator)
  • buybackBps → BuybackMiner (converts ETH to BKC rewards)
  • treasuryBps → Treasury
  • customBps → Module-specific (e.g., prize pool for FortunePool)
User performs action (swap, stake, buy NFT, etc.)
↓
ETH fee sent to BackchainEcosystem
↓
10% → Tutor (or burned)
↓
Remaining 90% split by module config:
↓ ↓ ↓ ↓
Operator  |  Buyback  |  Treasury  |  Custom
Zero leakage guarantee: All percentages are immutable per module. Every wei is accounted for — there are no unallocated funds and no admin-controlled drains.
Section 5

Growth Systems

Tutor System

The Tutor system is Backchain's on-chain referral program. Any user can become a tutor by sharing their referral link, creating a permanent economic relationship with their referrals.

  • Link format: backcoin.org/#dashboard?ref=0xADDRESS
  • First-time setup: gasless (server relayer pays)
  • Tutor earns: 10% of student's ETH fees + 5% of student's BKC claims
  • Relationship: stored on-chain, permanent (but student can change tutor for 0.0001 ETH)
  • No tutor: the 10% cut is burned instead

Operator System

The Operator system allows any developer to build on Backchain and earn commissions. There is no registration, no API key, and no approval needed — it is fully permissionless.

  • Integration: set via window.BACKCHAIN_OPERATOR or localStorage.setItem('bkc_operator', addr)
  • Every transaction includes the operator address in contract calls
  • Operator earns: 10–20% of every fee generated from their app
  • ETH accumulates in pendingEth[operator], withdrawable at any time
  • No registration, no API key, no approval needed
Builder-friendly by design: The Operator system means any developer can fork the frontend, customize the UI, and earn commissions from every transaction their users make — without needing permission from anyone.

Dual Provider Pattern

The frontend maintains two independent sets of contract instances at all times:

  • Signer contracts — connected to MetaMask, used exclusively for write transactions (signing)
  • Public contracts — connected to Alchemy RPC, used for all read operations without triggering MetaMask popups

This separation eliminates MetaMask rate-limiting errors (-32002) and ensures background data polling never interrupts the user experience.

Section 7

Security Model

Backchain's security model prioritizes immutability and trustlessness over administrative flexibility. Once deployed, the contracts cannot be altered, paused, or censored by anyone.

  • No admin keys on deployed contracts
  • No pause/unpause functions
  • No token blacklists
  • No proxy upgrades — immutable deployment
  • All contracts verified on Arbiscan

Progressive Governance

While the contracts themselves are immutable, governance parameters follow a progressive decentralization path managed by BackchainGovernance.sol:

Phase 1 Admin — current phase (testnet)
Phase 2 Multisig — shared control
Phase 3 Timelock — delayed execution
Phase 4 Full DAO — community governance
